# 需求分析 (Requirements Analysis)

## 1. 业务场景描述 (Scenarios)
- 用户通过账号密码登录系统，只能看到自己的联系人、关系标签与事项数据。
- 用户管理联系人资料，包括新增、查询、修改、加入黑名单、恢复、头像维护以及关系标签绑定/解绑。
- 用户管理关系标签，支持新增、编辑、删除和自定义颜色，并可在联系人列表中按标签筛选。
- 用户围绕联系人记录事项，支持新增、完成、取消、搜索、分页、状态筛选与时间排序。
- 用户在首页看板查看个性化问候（展示“早上好/下午好”与当前日期）和自己的本地天气，天气数据由后端根据请求 IP 自动定位并兜底。在联系人详情页，系统根据联系人地址模糊检索其所在城市的近 3 天天气预报，支持行内折叠滑出，为用户提供沟通关怀支持；当客户地址缺失时，支持一键引导补全。
- 用户可在联系人表单与个人设置页上传头像，系统支持文件类型（jpg/jpeg/png/webp）与大小（最大2MB）校验，提供失败提示并保留旧头像回显。
- 用户在基础表单能力之外，通过 Contact Agent 进行自然语言交互。智能助手已支持独立页面下的联系人/事项查询，以及“创建事项”写操作的预确认与二次确认落库闭环（TASK-010）。

## 2. 用户角色与用例 (User Roles & Use Cases)
| 角色名称 | 职责描述 | 核心用例 |
|---|---|---|
| 普通用户 | 管理自己的联系人、标签和事项 | 注册、登录、维护联系人、维护关系标签、维护事项、查看看板、使用 Agent、使用天气关怀组件 |
| 系统管理员（预留） | 后续扩展的系统运维与演示数据管理 | 用户管理、系统配置、演示环境维护 |

## 3. 系统核心业务流 (Core Business Flow)
1. 用户注册或登录后获得 JWT，并以该身份访问联系人、关系标签、事项、看板与上传接口。
2. 联系人、关系标签与事项按用户归属隔离；联系人可切换为正常或黑名单状态，但不做物理删除。
3. 关系标签删除时，自动在应用层级联清理 `contact_tag` 关联记录，不删除或修改联系人本身。
4. 头像文件（联系人及用户头像）存储在本地 `./uploads` 目录中，支持格式及 2MB 大小校验，数据库只保存文件访问相对路径，并对上传异常提供失败提示与原图预览兜底。
5. Contact Agent 解析用户意图后，对于查询类请求直接复用业务 Service 返回结果；对于“创建事项”写操作请求，进入“预处理 -> 返回确认卡片 -> 用户确认提交 -> 真实业务落库”的受控写操作确认流。对于其他写请求，仍提示不支持。
6. 天气组件核心业务流：前端发起请求后，后端 Weather 模块代理请求第三方和风天气 API。后端自动缓存同一城市的请求结果 2 小时，避免高频调用；在联系人地址模糊检索时，先通过 GeoAPI 将详细地址映射为城市 ID，再换取 3 天天气预报。

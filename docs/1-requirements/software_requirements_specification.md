# 软件需求规格说明书 (SRS)

## 1. 引言

### 1.1 编写目的

本文档用于明确 Personal CRM 智能联系人管理平台的业务目标、功能范围、非功能需求和验收标准，为后续概要设计、详细设计、编码实现、测试验收和部署上线提供统一基线。

### 1.2 项目背景

本项目面向个人联系人管理场景，目标是将联系人资料、联系人事项、黑名单、头像资料和轻量 Contact Agent 能力整合到同一个前后端分离系统中。系统既要满足课程实习指导书的基础验收要求，也要具备一定工程化与作品集展示价值。

### 1.3 术语说明

| 术语 | 说明 |
|---|---|
| 用户 | 登录系统的个人账号 |
| 联系人 | 由用户维护的联系人资料 |
| 黑名单 | 被用户标记为屏蔽状态的联系人 |
| 事项 | 围绕联系人记录的待办事件 |
| Contact Agent | 提供自然语言查询与受控写操作确认的辅助模块 |

## 2. 项目目标与范围

### 2.1 建设目标

1. 实现账号登录鉴权与用户数据隔离。
2. 实现联系人资料的新增、查询、修改、黑名单管理和头像维护。
3. 实现联系人事项的新增、筛选、完成、取消和逾期判断。
4. 实现首页看板统计与基础可视化展示。
5. 在基础链路稳定后，扩展 Contact Agent 的自然语言交互能力。

### 2.2 项目范围

本期范围包含：

1. 登录与当前用户身份获取。
2. 联系人管理（支持标签与黑名单管理）。
3. 联系人事项管理（支持状态变更与逾期判断）。
4. 用户头像与联系人头像上传。
5. 基于 Redis 缓存的多维数据看板。
6. Agent 查询与写操作确认链路。
7. 基于 Redis + Resend SDK 的安全邮箱验证码发送、注册激活与密码找回。

本期暂不包含：

1. 多角色后台管理。
2. 批量导入导出。
3. 手机短信或即时消息推送。
4. 微服务分布式服务拆分。
5. 生产级 OSS 强依赖。

## 3. 用户角色

| 角色 | 描述 | 权限范围 |
|---|---|---|
| 普通用户 | 系统主要使用者 | 仅可访问本人联系人、事项、头像和 Agent 记录 |
| 系统管理员（预留） | 后续扩展角色 | 本期不实现独立管理后台 |

## 4. 业务流程

### 4.1 登录业务流

1. 用户输入账号与密码。
2. 后端校验账号状态与密码摘要。
3. 登录成功后签发 JWT。
4. 前端持久化 token，并在后续请求中携带。

### 4.2 联系人管理业务流

1. 用户登录后进入联系人列表。
2. 用户可新增、查看、修改联系人资料。
3. 用户可将联系人加入黑名单或恢复正常状态。
4. 系统按当前登录用户进行数据归属隔离。

### 4.3 事项管理业务流

1. 用户选择联系人创建事项。
2. 系统保存事项时间、内容、状态和优先级。
3. 用户可将待完成事项更新为已完成或已取消。
4. 逾期状态由系统动态判断，不单独存储。

### 4.4 Agent 业务流

1. 用户输入自然语言请求。
2. Agent 解析意图和参数。
3. 查询类请求直接调用既有业务服务返回结果。
4. 写操作类请求先生成预览，再等待用户确认后执行。

## 5. 功能需求

### 5.1 认证与安全验证模块

1. 用户可以通过账号和密码进行登录。
2. 登录成功后，系统应生成并返回 JWT Token 作为后续 API 请求的会话凭据。
3. 系统应支持获取当前登录用户的详细个人资料（包含头像与邮箱、手机等）。
4. 系统应支持用户注册功能，注册阶段必须强制提供有效的电子邮箱，并校验 6 位随机数邮件验证码。
5. 系统应通过 Redis 实施发信限流与验证码防护规则（验证码 300 秒有效期、60 秒防频繁锁、5 次输错后作废计数锁）。
6. 系统应支持未登录状态下通过邮箱验证码进行找回/重置密码。
7. 系统应支持登录态下通过验证码修改绑定邮箱。
8. 未登录用户不得访问系统任何受保护业务接口（统一拦截并返回 401 状态）。

### 5.2 联系人模块

1. 用户可以查看联系人分页列表。
2. 用户可以按姓名、手机号、微信等关键词搜索联系人。
3. 用户可以新增联系人。
4. 用户可以修改联系人。
5. 用户可以将联系人加入黑名单。
6. 用户可以将黑名单联系人恢复为正常状态。
7. 用户只能查看和操作自己的联系人。

### 5.3 事项模块

1. 用户可以查看事项分页列表。
2. 用户可以按联系人、状态和时间区间筛选事项。
3. 用户可以新增事项。
4. 用户可以将待完成事项标记为已完成。
5. 用户可以将待完成事项标记为已取消。
6. 已完成和已取消事项不允许重复变更状态。

### 5.4 上传模块

1. 用户可以上传自己的头像。
2. 用户可以上传联系人的头像。
3. 系统应校验文件类型和大小。
4. 系统应保存文件路径并返回访问地址。

### 5.5 看板模块

1. 系统应统计联系人总数。
2. 系统应统计黑名单联系人数量。
3. 系统应统计待完成事项数量。
4. 系统应统计今日事项数量。
5. 系统应统计逾期事项数量。
6. 系统应支持基础图表数据接口。

### 5.6 Agent 模块

1. Agent 应支持自然语言查询联系人与事项。
2. Agent 应支持自然语言创建事项的预确认流程。
3. Agent 不得绕过既有 Service 直接写库。
4. Agent 的写操作必须记录操作日志。

## 6. 非功能需求

### 6.1 性能与资源控制需求

1. 常规分页与搜索查询接口在本地开发环境下应在 1.5 秒内返回。
2. 首页看板统计在普通数据规模下应在 3 秒内返回；开启 Redis 二级缓存后，命中缓存请求的接口响应时间应控制在 5ms 以内，大幅度削减数据库物理 IO。
3. 单图片文件上传大小限制在 2MB 以内。
4. 前端应用应做好资源生命周期管理，在 ECharts 仪表盘组件注销或路由切换时，必须显式清空并销毁图表实例变量（置为 null），以防止闭包造成的常驻内存泄露（Memory Leak）。
5. 后端 AI 会话采用外部化分布式 Redis TTL 自动管理机制，JVM 常驻会话内存开销降为 0。

### 6.2 安全需求

1. 密码必须以摘要形式存储。
2. 所有受保护接口必须校验 JWT。
3. 用户数据必须按当前登录用户隔离。
4. 接口必须使用参数化查询，避免 SQL 注入。
5. Agent 写操作必须经过二次确认。

### 6.3 可维护性需求

1. 前后端分层明确，职责清晰。
2. 接口响应格式统一。
3. 文档、数据库设计、接口契约和代码实现必须保持同步。

### 6.4 可用性需求

1. 页面操作流程应清晰直观。
2. 表单输入应具备基础校验与错误提示。
3. 关键业务异常应返回明确提示信息。

## 7. 外部接口需求

### 7.1 前后端接口

系统采用 RESTful API，统一前缀 `/api/v1`，统一响应格式为 `{ code, message, data }`。

### 7.2 数据库接口

系统使用 MySQL 8 保存结构化数据，开发阶段通过 `schema.sql` 和 `data.sql` 进行初始化。

### 7.3 文件存储接口

本期默认使用本地文件系统保存头像文件，后续预留 OSS 扩展能力。

## 8. 验收标准

### 8.1 指导书基线验收

1. 数据库至少包含 `sys_user`、`user_avatar`、`contact`、`contact_avatar`、`contact_todo` 五张基线表。
2. 系统应支持联系人信息管理。
3. 系统应支持联系人事项管理。
4. 系统应支持联系人屏蔽或黑名单管理。

### 8.2 项目实现验收

1. 用户能够通过邮箱激活完成注册登录，并严格按租户隔离访问本人数据。
2. 联系人模块支持新增、查询、编辑修改、状态切换以及逻辑删除操作。
3. 事项模块支持新增、查询、完成、取消和逻辑删除操作。
4. 邮箱模块支持发送验证码、安全校验、注册激活、找回密码与更换邮箱功能，校验规则对齐防爆破和防刷频限流标准。
5. 看板模块支持二级缓存毫秒级读取，并在数据写操作后自动清空缓存一致。
6. 上传模块能完成头像文件的尺寸校验、格式限制、上传持久化和相对路径回显。
7. Agent 模块完成自然语言查询、多轮澄清补槽机制、以及涉及创建待办等写操作的二次确认控制流闭环。

## 9. 约束与风险

1. 本项目为课程实习项目，应优先保证主链路稳定，不盲目引入高复杂度中间件。
2. Agent 属于增强功能，不应影响基线验收。
3. 若时间紧张，应优先完成登录、联系人、事项、看板和上传，再补 Agent。

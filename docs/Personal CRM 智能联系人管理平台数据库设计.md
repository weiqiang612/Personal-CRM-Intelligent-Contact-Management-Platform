# Personal CRM 智能联系人管理平台数据库设计

## 1. 数据库设计概述

本系统采用 MySQL 8 作为关系型数据库，用于持久化保存用户、联系人、联系人事项、头像信息，以及在扩展阶段使用的联系人标签和 Contact Agent 操作记录等数据。

为了同时满足课程验收和项目扩展两个目标，本数据库设计分为两层：

1. **指导书基线设计**：先满足实习指导书要求的 5 张基础表。
2. **项目扩展设计**：在指导书基线之上，增加标签和 Agent 审计等增强能力。

数据库设计围绕以下目标展开：

1. 保证用户数据隔离，不同用户只能访问本人账号下的数据。
2. 支持联系人信息的新增、查询、修改、加入黑名单和恢复。
3. 支持联系人事项的创建、查询、完成、取消和逾期判断。
4. 支持头像上传后的路径保存与前端回显。
5. 在满足指导书要求的基础上，扩展标签和 Contact Agent 能力。
6. 支持列表分页、搜索、筛选、排序和数据看板统计。

### 1.1 指导书基线表

根据《计算机专业实习指导书》，数据库基线应包含以下 5 张表：

```text
sys_user          用户信息表
user_avatar       用户图片信息表
contact           联系人信息表
contact_avatar    联系人图片信息表
contact_todo      联系人事项信息表
```

### 1.2 项目扩展表

结合本项目实际需求，在基础表完成后，再扩展以下 4 张增强表：

```text
tag                  联系人标签表
contact_tag          联系人标签关联表
agent_operation_log  Agent 操作记录表
activity_log         联系人活动轨迹日志表
```

说明：

1. 指导书要求的 5 张表是课程验收的核心基线。
2. 标签、Agent 操作记录和活动日志属于增强设计，不影响指导书基础功能验收。
3. 本文先描述基线表，再描述扩展表。

## 2. 数据库实体关系设计

### 2.1 指导书基线实体关系

```text
用户 1 —— N 联系人
用户 1 —— 1 用户头像记录（也可按实现保留为 1 —— N）
联系人 1 —— 1 联系人头像记录（也可按实现保留为 1 —— N）
联系人 1 —— N 联系人事项
```

### 2.2 扩展实体关系

```text
用户 1 —— N 标签
联系人 N —— N 标签
用户 1 —— N Agent 操作记录
联系人 1 —— N 活动日志
```

其中，`user_id` 是系统数据隔离的核心字段。联系人、事项、标签、头像记录、Agent 操作记录以及活动日志等业务表均需要保存或间接关联当前用户 ID，防止用户访问其他账号下的数据。

## 3. 指导书基线数据表设计

## 3.1 用户信息表 sys_user

该表对应指导书中的“用户信息表”，用于保存系统登录账号信息和密码摘要。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 用户主键 ID |
| user_id | char(10) | 非空，唯一 | 用户业务编号，对应指导书中的用户 ID |
| username | varchar(50) | 非空，唯一 | 登录账号 |
| password_hash | varchar(255) | 非空 | 密码加密摘要 |
| status | tinyint | 非空，默认 0 | 用户状态，0 正常，1 禁用，2 待验证 |
| email | varchar(100) | 可空 | 电子邮箱 |
| email_verified | tinyint(1) | 非空，默认 0 | 邮箱是否已验证，0 否，1 是 |
| email_verified_at | datetime | 可空 | 邮箱激活验证时间 |
| phone | varchar(30) | 可空 | 手机号码 |
| created_at | datetime | 非空 | 创建时间 |
| updated_at | datetime | 非空 | 更新时间 |

设计说明：

1. 指导书要求用户信息表至少包含用户 ID 和密码，本设计在此基础上保留 `user_id` 业务编号，并额外引入 `id` 作为工程实现主键。
2. 密码字段保存加密后的摘要，不直接保存明文密码。
3. `username` 用于登录，`user_id` 用于对齐指导书中的数据结构说明。
4. `status` 字段用于后续扩展账号禁用及验证挂起能力（0 激活正常，2 待注册激活）。
5. `email`、`email_verified`、`email_verified_at` 和 `phone` 字段为功能扩展，用于支持账号注册激活、验证码频率锁、个人资料修改邮箱和手机号码等账户安全功能。

该表对应指导书中的“用户图片信息表”，用于保存登录用户头像信息。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 记录主键 ID |
| pic_id | char(10) | 非空，唯一 | 图片业务编号，对应指导书中的图片 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| file_name | varchar(255) | 非空 | 图片文件名 |
| file_path | varchar(255) | 可空 | 文件存储路径 |
| access_url | varchar(255) | 可空 | 前端访问路径 |
| created_at | datetime | 非空 | 上传时间 |

设计说明：

1. 指导书要求此表至少包含 `user_id`、`pic_id`、`pic_name`，本设计保留其核心语义，并将 `pic_name` 扩展为更实用的 `file_name`。
2. 用户头像不作为本项目核心功能，但保留此表可完整满足指导书数据库结构要求。
3. 若课程实现阶段不展示用户头像，也可以保留表结构而不优先开发对应页面能力。

## 3.3 联系人信息表 contact

该表对应指导书中的“联系人信息表”，是系统核心业务表。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 联系人主键 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| ct_id | char(10) | 非空，唯一 | 联系人业务编号，对应指导书中的联系人 ID |
| name | varchar(50) | 非空 | 联系人姓名 |
| address | varchar(255) | 可空 | 联系人地址 |
| postcode | varchar(20) | 可空 | 联系人邮编 |
| qq | varchar(30) | 可空 | 联系人 QQ |
| wechat | varchar(50) | 可空 | 联系人微信 |
| email | varchar(100) | 可空 | 联系人邮箱 |
| gender | tinyint | 可空 | 性别，0 未知，1 男，2 女 |
| birthday | date | 可空 | 联系人生日 |
| phone | varchar(30) | 可空 | 联系人电话 |
| remarks | varchar(500) | 可空 | 备注 |
| status | tinyint | 非空，默认 0 | 联系人状态，0 正常，1 黑名单 |
| deleted | tinyint | 非空，默认 0 | 逻辑删除标志，0 正常，1 已删除 |
| created_at | datetime | 非空 | 创建时间 |
| updated_at | datetime | 非空 | 更新时间 |

设计说明：

1. 指导书要求联系人表至少包含用户 ID、联系人 ID、姓名、地址、邮编、QQ、微信、邮箱、性别、出生日期、电话、状态等字段，本设计全部保留。
2. `status` 字段用于实现指导书中“屏蔽联系人”的要求，0 代表正常，1 代表黑名单。
3. 联系人黑名单不单独建表，而是通过 `status` 字段区分，符合指导书对“可屏蔽不物理删除”的提示。
4. `deleted` 字段为功能扩展，配合 MyBatis-Plus 逻辑删除处理器，使系统删除联系人时保留历史记录以维护事项和日志的一致性。
5. `birthday` 采用 `date` 比使用字符串更利于排序和后续统计。

## 3.4 联系人图片信息表 contact_avatar

该表对应指导书中的“联系人图片信息表”，用于保存联系人头像信息。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 记录主键 ID |
| pic_id | char(10) | 非空，唯一 | 图片业务编号，对应指导书中的图片 ID |
| contact_id | char(10) | 非空，索引 | 联系人业务编号 |
| file_name | varchar(255) | 非空 | 图片文件名 |
| file_path | varchar(255) | 可空 | 文件存储路径 |
| access_url | varchar(255) | 可空 | 前端访问路径 |
| created_at | datetime | 非空 | 上传时间 |

设计说明：

1. 指导书要求该表至少包含 `ct_id`、`pic_id`、`pic_name`，本设计保留业务含义并扩展出存储路径和访问路径。
2. 指导书从课程角度可理解为“一位联系人对应一张当前头像”；工程实现时，也可以允许一个联系人保留多次上传历史，但当前阶段建议界面上只使用最近一张或当前指定头像。
3. 如果后续要支持头像历史记录，建议再增加 `is_current` 字段，但不作为指导书基线的必需项。

## 3.5 联系人事项信息表 contact_todo

该表对应指导书中的“联系人事项信息表”，用于保存联系人相关事项。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 事项主键 ID |
| matter_id | char(10) | 非空，唯一 | 事项业务编号，对应指导书中的事项 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| contact_id | char(10) | 非空，索引 | 联系人业务编号 |
| todo_time | datetime | 非空，索引 | 事项时间 |
| content | varchar(500) | 非空 | 事项内容 |
| status | tinyint | 非空，默认 0 | 事项状态，0 待完成，1 已取消，2 已完成 |
| priority | tinyint | 非空，默认 0 | 优先级，0 普通，1 重要，2 紧急 |
| completed_at | datetime | 可空 | 完成时间 |
| cancelled_at | datetime | 可空 | 取消时间 |
| deleted | tinyint | 非空，默认 0 | 逻辑删除标志，0 正常，1 已删除 |
| created_at | datetime | 非空 | 创建时间 |
| updated_at | datetime | 非空 | 更新时间 |

设计说明：

1. 指导书要求事项表至少包含联系人 ID、事项 ID、事项时间、事项内容、事项状态，本设计全部保留。
2. 为了便于查询当前用户所有事项与看板统计，本设计冗余保存 `user_id`。
3. 事项状态严格对齐指导书和需求文档：`0 待完成，1 已取消，2 已完成`。
4. `priority` 是项目扩展字段，但不影响指导书基础功能。
5. 事项逾期不单独存储，而是通过 `todo_time < 当前时间` 且 `status = 0` 动态判断。
6. `deleted` 字段支持逻辑删除，确保用户删除待办时，其操作记录和活动流日志等仍具有数据参照完整性。

## 4. 项目扩展数据表设计

## 4.1 标签表 tag

标签表用于保存用户自定义的联系人标签，例如同学、老师、朋友、客户、重要联系人等。该能力来自项目需求文档中的增强功能，不属于指导书基础表。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 标签 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| name | varchar(50) | 非空 | 标签名称 |
| color | varchar(20) | 可空 | 标签颜色 |
| created_at | datetime | 非空 | 创建时间 |
| updated_at | datetime | 非空 | 更新时间 |

设计说明：

1. 标签属于增强功能，在指导书基础功能完成后再实现。
2. 同一个用户下标签名称不允许重复。
3. 不同用户之间可以创建同名标签。

## 4.2 联系人标签关联表 contact_tag

联系人标签关联表用于维护联系人和标签之间的多对多关系。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 关联 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| contact_id | char(10) | 非空，索引 | 联系人业务编号 |
| tag_id | bigint | 非空，索引 | 标签 ID |
| created_at | datetime | 非空 | 创建时间 |

设计说明：

1. 一个联系人可以绑定多个标签。
2. 一个标签可以绑定多个联系人。
3. 建议对 `contact_id` 和 `tag_id` 建立联合唯一约束，避免重复绑定。

## 4.3 Agent 操作记录表 agent_operation_log

Agent 操作记录表用于保存 Contact Agent 的自然语言输入、意图识别结果、解析参数、确认状态和执行结果。该表用于增强系统可追溯性，不属于指导书基线表。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 记录 ID |
| user_id | char(10) | 非空，索引 | 用户业务编号 |
| user_input | text | 非空 | 用户原始输入 |
| intent | varchar(100) | 可空 | 识别出的意图 |
| parsed_params | json | 可空 | 解析出的结构化参数 |
| need_confirm | tinyint | 非空，默认 0 | 是否需要确认，0 否，1 是 |
| confirmed | tinyint | 非空，默认 0 | 用户是否已确认，0 否，1 是 |
| action_type | varchar(100) | 可空 | 操作类型 |
| execution_status | tinyint | 非空，默认 0 | 执行状态，0 待确认，1 成功，2 失败，3 已取消 |
| execution_result | json | 可空 | 执行结果 |
| error_message | varchar(500) | 可空 | 错误信息 |
| created_at | datetime | 非空 | 创建时间 |

设计说明：

1. 查询类 Agent 操作可以直接返回结果，但仍可以保存操作记录。
2. 写操作必须先记录解析结果和影响范围，等待用户确认后再执行。
3. `parsed_params` 和 `execution_result` 采用 JSON，便于保存灵活结构。

## 4.4 活动日志表 activity_log

活动日志表用于保存联系人以及关联事项的增删改等核心活动痕迹，为前端时间线提供真实的数据源。

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | bigint | 主键，自增 | 自增主键 ID |
| activity_id | varchar(64) | 非空，唯一 | 活动业务编号 |
| user_id | varchar(64) | 非空 | 用户业务编号 |
| contact_id | varchar(64) | 非空 | 联系人业务编号 |
| event_type | varchar(32) | 非空 | 事件类型枚举，如 TODO_COMPLETED |
| title | varchar(128) | 非空 | 动作总结标题 |
| description | text | 可空 | 活动详细内容描述 |
| occurred_at | datetime | 非空 | 活动发生时间 |
| created_at | datetime | 非空 | 记录创建时间 |

设计说明：

1. 采用 `user_id` 和 `contact_id` 联合索引，保证多租户越权防范，并加速详情页轨迹查询。
2. 事件类型以 `eventType` 的枚举值在 Java 代码中解析，并在前端做图标与文字映射。
3. 历史活动记录一经写入不提供修改接口，删除联系人时同步逻辑屏蔽其活动记录。

## 5. 状态字段设计

## 5.1 用户状态

| 状态值 | 含义 |
|---|---|
| 0 | 正常 (ACTIVE) |
| 1 | 禁用 (DISABLED) |
| 2 | 待验证 (UNVERIFIED) |

## 5.2 联系人状态

| 状态值 | 含义 |
|---|---|
| 0 | 正常 |
| 1 | 黑名单 |

## 5.3 事项状态

| 状态值 | 含义 |
|---|---|
| 0 | 待完成 |
| 1 | 已取消 |
| 2 | 已完成 |

事项状态流转规则如下：

```text
待完成 -> 已完成
待完成 -> 已取消
已完成 -> 不允许重复修改
已取消 -> 不允许重复修改
```

如后续需要扩展恢复功能，可以增加状态恢复接口，但恢复操作需要二次确认。

## 5.4 事项优先级

| 状态值 | 含义 |
|---|---|
| 0 | 普通 |
| 1 | 重要 |
| 2 | 紧急 |

## 5.5 Agent 执行状态

| 状态值 | 含义 |
|---|---|
| 0 | 待确认 |
| 1 | 执行成功 |
| 2 | 执行失败 |
| 3 | 用户取消 |

## 5.6 活动事件类型 (Activity Event Type)

活动日志中 `event_type` 字段存储的字符串枚举值与业务行为对应关系如下：

| 事件类型枚举 | 触发业务行为 |
|---|---|
| CONTACT_CREATED | 新增联系人成功时 |
| CONTACT_UPDATED | 修改联系人基本信息成功时 |
| CONTACT_DELETED | 逻辑删除联系人成功时 |
| CONTACT_BLACKLISTED | 将联系人拉入黑名单时 |
| CONTACT_RESTORED | 从黑名单恢复联系人时 |
| TAG_CHANGED | 变更联系人的标签绑定关系时 |
| TODO_CREATED | 创建该联系人下的提醒事项成功时 |
| TODO_COMPLETED | 将关联事项标记为已完成时 |
| TODO_CANCELLED | 将关联事项标记为已取消时 |
| TODO_DELETED | 逻辑删除关联事项成功时 |

## 6. 索引设计

为了支持搜索、筛选、排序、分页和数据看板统计，需要设计合理索引。

## 6.1 用户表索引

```text
唯一索引：user_id
唯一索引：username
```

用途：

1. 登录时根据账号查询用户。
2. 防止重复注册或重复初始化账号。
3. 保留指导书风格的用户业务编号。

## 6.2 联系人表索引

```text
普通索引：user_id, status
普通索引：user_id, name
普通索引：user_id, phone
普通索引：user_id, birthday
普通索引：user_id, created_at
```

用途：

1. 查询当前用户的正常联系人列表。
2. 查询当前用户的黑名单联系人列表。
3. 按姓名或手机号搜索联系人。
4. 按生日或创建时间排序。

## 6.3 联系人事项表索引

```text
普通索引：user_id, status, todo_time
普通索引：user_id, contact_id
普通索引：user_id, todo_time
普通索引：contact_id, status
```

用途：

1. 查询今日事项。
2. 查询逾期事项。
3. 查询未来 7 天事项。
4. 查询某个联系人的事项列表。
5. 支持数据看板统计待完成、已完成、逾期事项数量。

说明：

1. 事项内容搜索在课程项目阶段可以先通过普通模糊查询实现。
2. 若后续数据量增大，再考虑全文索引或专项优化。

## 6.4 标签表索引

```text
唯一索引：user_id, name
```

## 6.5 联系人标签关联表索引

```text
唯一索引：contact_id, tag_id
普通索引：tag_id, contact_id
普通索引：user_id, tag_id
```

## 6.6 Agent 操作记录表索引

```text
普通索引：user_id, created_at
普通索引：user_id, intent
```

## 6.7 活动日志表索引

```text
唯一索引：activity_id
普通索引：user_id, contact_id, occurred_at
```

用途：

1. 加速联系人详情页操作轨迹流时间线查询（过滤归属且倒序排列）。
2. 提供活动流水记录唯一性保障。

## 7. 数据完整性设计

## 7.1 主键策略

本项目在工程实现上统一使用 `bigint` 自增主键作为真实主键，便于后端实体映射和接口传参；同时为对齐指导书，在核心基础表中保留 `user_id`、`ct_id`、`pic_id`、`matter_id` 等业务编号字段。

## 7.2 外键关系

系统逻辑上存在如下关系：

```text
user_avatar.user_id -> sys_user.user_id
contact.user_id -> sys_user.user_id
contact_avatar.contact_id -> contact.ct_id
contact_todo.user_id -> sys_user.user_id
contact_todo.contact_id -> contact.ct_id
tag.user_id -> sys_user.user_id
contact_tag.user_id -> sys_user.user_id
contact_tag.contact_id -> contact.ct_id
contact_tag.tag_id -> tag.id
agent_operation_log.user_id -> sys_user.user_id
activity_log.user_id -> sys_user.user_id
activity_log.contact_id -> contact.ct_id
```

实际开发中可以根据项目复杂度选择是否使用数据库物理外键。为了降低开发和数据初始化复杂度，本项目采用业务层校验为主、数据库索引约束为辅的方式。

## 7.3 用户数据隔离

用户数据隔离是本系统数据库设计的核心要求。联系人、事项、标签、头像记录和 Agent 操作记录均需要关联 `user_id`。

后端查询时必须携带当前登录用户 ID，例如：

```text
查询联系人：where user_id = 当前用户ID and status = 0
查询事项：where user_id = 当前用户ID
查询标签：where user_id = 当前用户ID
```

在修改、完成、取消、加入黑名单等操作前，后端必须校验目标数据是否属于当前用户。

## 7.4 软删除与状态控制

本系统引入逻辑删除机制，在删除联系人 (`contact`) 和事项 (`contact_todo`) 时不进行物理删除，而是将其对应的 `deleted` 字段置为 1。

- **逻辑删除联系人**：更新 `deleted = 1` 时，系统在应用层级联删除其在 `contact_tag` 中的关联记录，但保留其历史事项以供历史归档和看板统计。
- **逻辑删除事项**：更新 `deleted = 1`，逻辑屏蔽该事项在列表及 AI 查询中的展现。
- **MyBatis-Plus 自动拦截**：通过在实体类中使用 `@TableLogic` 注解配置，实现底层的自动软删除过滤。

对于联系人黑名单，也不采用物理删除，而是通过 `contact.status` 状态字段 (0 正常，1 黑名单) 控制。

## 8. Redis 缓存与分布式内存数据结构设计

系统在后端集成了 Redis，主要用于提高数据看板访问性能、维护无状态的 AI 智能会话、以及实现防暴力破解和频繁限制的邮箱验证机制。

### 8.1 邮箱安全验证码与防爆破模型

系统针对发送至邮箱的安全激活码设置三级防范存储结构：
1. **验证码信息 (Code)**：
   - Key 结构：`email:code:{purpose}:{email}`
   - 值：SHA-256 加密后的验证码摘要
   - TTL：300 秒（过期自动废弃）
2. **防频繁发送限制锁 (Send Rate Limit)**：
   - Key 结构：`email:send:{purpose}:{email}`
   - 值：`1`
   - TTL：60 秒（起防刷频限制作用，60秒内拒绝再次发送）
3. **安全防爆破输错锁定锁 (Attempt Count)**：
   - Key 结构：`email:attempt:{purpose}:{email}`
   - 值：当前校验失败的累加计数（初始为 0，最大为 5）
   - TTL：300 秒（与验证码同周期失效。累计至 5 次后，系统完全锁定该验证码的校验行为，直至其失效并提示用户重新发送）

### 8.2 首页数据看板二级缓存

为应对可能存在的高频仪表盘刷新访问，首页看板统计指标引入 Redis 缓存加速设计：
- **存储结构**：Redis Hash
- **Key 结构**：`dashboard:cache:{userId}`（以用户业务编号物理隔离）
- **字段 (Fields)**：存储看板四个维度数值 VO 对象的 JSON 字符串
- **缓存策略 (TTL)**：300 秒（5分钟）
- **被动删除机制**：当该用户名下的联系人、事项或标签发生任何写操作（新增、更新、状态切换、逻辑删除）时，后端通过拦截器/AOP 机制一键删除该用户的 `dashboard:cache:{userId}` 键值，保证数据看板强一致性。

### 8.3 Contact Agent 分布式多轮对话会话状态

为了使 Agent服务保持分布式无状态以及抗单点崩溃自愈，系统将会话上下文托管在 Redis 中：
- **存储结构**：Redis String
- **Key 结构**：`agent:session:{sessionId}`
- **值**：序列化为 JSON 字符串的 `AgentSessionState`（包括已填充的槽位信息、个人对话历史等）
- **会话策略 (TTL)**：600 秒（10分钟）自动过期。会话一旦超出此时间没有更新，自动注销释放资源，避免 JVM 常驻内存浪费。

## 9. 数据库设计特点

本数据库设计具有以下特点：

1. 先满足指导书要求的 5 张基础表，再在其上进行标签、活动日志和 Agent 审计扩展。
2. 通过 `user_id` 实现用户数据隔离，降低水平越权风险。
3. 通过 `status` 字段与 `deleted` 字段实现联系人状态管理与逻辑删除，避免物理删除带来的数据丢失，保护了活动日志的引用一致性。
4. 基于 Redis 集中设计了限流冷却锁、验证码尝试次数锁定机制，提升了账号注册与密码重置的防爆破安全性。
5. 设计看板缓存和 Agent 分布式会话存储模型，在降低数据库压力的同时实现服务器端无状态化。
6. 在保留指导书业务编号字段的同时，引入更适合工程实现的自增主键方案。

## 10. 本章小结

本章对 Personal CRM 智能联系人管理平台进行了数据库设计。设计以指导书要求为基线，先落实用户、用户图片、联系人、联系人图片、联系人事项 5 张基础表，再结合项目需求扩展联系人标签、活动轨迹流、Redis 缓存结构和 Agent 操作记录等增强能力。

该设计既能够满足课程实习对数据库结构的基本要求，也能支撑本项目在联系人管理、黑名单管理、事项提醒、头像上传、数据看板、邮箱安全防爆破和 Contact Agent 智能交互方面的扩展实现。

# Personal CRM 智能联系人管理平台需求分析

## 1. 项目背景

随着个人学习、实习、社交和工作场景的不断扩展，用户需要维护的联系人数量逐渐增多，联系人之间的关系类型、沟通记录、待办事项和提醒任务也变得更加复杂。传统通讯录通常只支持姓名、电话、邮箱等基础信息的保存，缺少对联系人事项、关系标签、提醒任务和智能检索的统一管理能力，难以满足用户在学习、实习、求职和日常协作场景下的精细化联系人管理需求。

为解决上述问题，本项目设计并实现一套 Personal CRM 智能联系人管理平台。系统以联系人信息管理为核心，围绕联系人资料维护、黑名单管理、事项提醒、标签分类、搜索筛选、头像上传和移动端适配等功能，构建一个完整的在线信息管理系统。在此基础上，系统引入轻量级 Contact Agent 智能助手，支持自然语言创建事项、查询联系人、查询待办任务和生成今日事项摘要，从而提升系统的交互效率和使用体验。

## 2. 项目目标

本项目的总体目标是实现一个具备完整业务链路、良好交互体验和一定智能化能力的联系人管理系统。系统需要支持用户登录、联系人增删改查、联系人详情维护、黑名单管理、事项提醒管理、头像上传、搜索筛选排序分页、移动端适配等基础功能，并在基础功能稳定运行的前提下，扩展自然语言交互能力。

具体目标如下：

1. 实现基于账号密码的用户登录功能，确保不同用户只能访问自己的联系人和事项数据。
2. 实现联系人信息的新增、查询、修改、屏蔽、恢复等完整管理能力。
3. 实现联系人头像上传、头像预览和头像更新功能。
4. 实现联系人事项的添加、查询、取消、完成和状态筛选功能。
5. 实现联系人列表、黑名单列表和事项列表的搜索、筛选、排序与分页功能。
6. 实现 PC 浏览器端和移动端浏览器的响应式适配。
7. 实现数据看板，展示联系人数量、黑名单数量、待完成事项、今日事项、逾期事项等统计信息。
8. 实现轻量级 Contact Agent，支持自然语言创建事项、自然语言查询联系人/事项、今日待办总结和高风险操作二次确认。
9. 保证系统代码结构清晰，具备良好的可维护性、异常处理能力和参数校验能力。

## 3. 用户角色分析

本系统主要面向单个普通用户使用，用户登录后只能管理本人账号下的联系人和事项数据。系统角色相对简单，主要包括以下两类：

### 3.1 普通用户

普通用户是系统的主要使用者，拥有联系人管理、事项管理、黑名单管理、头像上传、智能助手使用等权限。

普通用户可以完成以下操作：

1. 登录系统。
2. 查看联系人列表。
3. 添加新的联系人。
4. 查看联系人详情。
5. 修改联系人资料。
6. 上传或更换联系人头像。
7. 将联系人加入黑名单。
8. 从黑名单中恢复联系人。
9. 查看联系人相关事项。
10. 添加联系人事项提醒。
11. 取消或完成事项。
12. 使用 Contact Agent 创建事项、查询联系人和总结今日待办。

### 3.2 系统管理员（可选扩展）

如果项目需要进一步扩展，可以增加系统管理员角色。系统管理员主要负责演示数据维护、用户管理和系统配置管理。但在本次实习项目中，管理员角色不是核心要求，可以作为后续扩展功能。

## 4. 功能性需求分析

## 4.1 用户登录模块

用户登录模块是系统入口。用户输入账号和密码后，系统对账号密码进行校验。校验通过后进入系统首页或联系人列表页面；校验失败则给出明确提示。

### 4.1.1 功能需求

1. 用户可以输入账号和密码。
2. 前端需要对账号和密码进行非空校验。
3. 后端需要校验账号是否存在、密码是否正确。
4. 登录成功后，系统保存当前用户身份信息。
5. 登录失败时，系统返回友好的错误提示。
6. 未登录用户不能访问联系人列表、联系人详情、事项管理等页面。

### 4.1.2 输入校验

1. 用户账号不能为空。
2. 用户密码不能为空。
3. 用户账号长度应符合数据库字段约束。
4. 用户密码长度应符合系统约定。

### 4.1.3 异常处理

1. 账号不存在时提示“账号不存在”或“账号或密码错误”。
2. 密码错误时提示“账号或密码错误”。
3. 系统异常时提示“系统繁忙，请稍后重试”。

## 4.2 联系人管理模块

联系人管理模块是系统核心模块，用于维护用户的联系人资料。联系人信息包括姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日、头像和联系人状态等。

### 4.2.1 联系人列表

联系人列表用于展示当前用户的正常联系人。用户可以在该页面查看联系人基本信息，并进行搜索、筛选、排序、分页、查看详情、添加联系人、加入黑名单和进入事项页面等操作。

功能需求如下：

1. 展示当前登录用户的正常联系人。
2. 支持分页展示联系人。
3. 支持根据姓名或手机号搜索联系人。
4. 支持根据性别筛选联系人。
5. 支持根据姓名、生日或创建时间进行排序。
6. 支持跳转到联系人详情页。
7. 支持跳转到添加联系人页面。
8. 支持将联系人加入黑名单。
9. 支持跳转到联系人事项页面。
10. 支持展示联系人头像、姓名、性别、电话等核心信息。

### 4.2.2 添加联系人

添加联系人页面用于录入新的联系人信息。用户填写联系人姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日等信息后，可以提交保存。

功能需求如下：

1. 支持输入联系人姓名。
2. 支持选择联系人性别。
3. 支持输入联系人电话。
4. 支持输入联系人邮箱。
5. 支持输入联系人 QQ。
6. 支持输入联系人微信。
7. 支持输入联系人地址。
8. 支持输入联系人邮编。
9. 支持选择联系人生日。
10. 支持上传联系人头像。
11. 支持添加成功后跳转回联系人列表页。
12. 支持添加失败时提示失败原因。

### 4.2.3 联系人详情

联系人详情页面用于查看和修改联系人完整资料。

功能需求如下：

1. 展示联系人姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日和头像。
2. 支持修改联系人资料。
3. 支持更换联系人头像。
4. 支持保存修改结果。
5. 支持跳转到该联系人的事项列表。
6. 支持返回联系人列表。

### 4.2.4 联系人黑名单

黑名单用于保存被屏蔽的联系人。加入黑名单后，该联系人不再出现在正常联系人列表中，但数据不会被物理删除。

功能需求如下：

1. 支持将正常联系人加入黑名单。
2. 支持查看黑名单联系人列表。
3. 支持搜索黑名单联系人。
4. 支持根据性别筛选黑名单联系人。
5. 支持分页展示黑名单联系人。
6. 支持查看黑名单联系人详情。
7. 支持将黑名单联系人恢复为正常联系人。

### 4.2.5 联系人标签扩展

为了增强 Personal CRM 的关系管理能力，系统可以为联系人增加标签功能。标签可以用于标识联系人关系类型，例如同学、老师、家人、朋友、客户、重要联系人等。

功能需求如下：

1. 支持创建联系人标签。
2. 支持给联系人绑定一个或多个标签。
3. 支持根据标签筛选联系人。
4. 支持在联系人详情中展示联系人标签。
5. 支持修改联系人标签。

该功能属于增强功能，在基础链路完成后实现。

## 4.3 联系人事项提醒模块

联系人事项提醒模块用于管理与联系人相关的待办事项。事项包括事项时间、事项内容、事项状态和所属联系人。

### 4.3.1 事项列表

事项列表用于展示联系人相关事项或当前用户所有事项。

功能需求如下：

1. 支持查看当前用户的所有事项。
2. 支持查看某个联系人的所有事项。
3. 支持根据事项内容搜索事项。
4. 支持根据事项状态筛选事项。
5. 支持根据事项时间升序或降序排序。
6. 支持分页展示事项。
7. 支持将事项标记为已完成。
8. 支持将事项标记为已取消。
9. 支持跳转到添加事项页面。
10. 支持展示事项所属联系人、事项时间、事项内容和事项状态。

### 4.3.2 添加事项

添加事项页面用于为联系人添加新的提醒事项。

功能需求如下：

1. 支持选择联系人。
2. 支持选择事项时间。
3. 支持输入事项内容。
4. 支持设置事项优先级。
5. 支持提交后保存为待完成事项。
6. 支持添加成功后返回事项列表。
7. 支持输入校验和错误提示。

### 4.3.3 事项状态管理

事项状态包括待完成、已取消和已完成。

功能需求如下：

1. 新增事项默认状态为待完成。
2. 用户可以将待完成事项标记为已完成。
3. 用户可以将待完成事项标记为已取消。
4. 已完成或已取消事项默认不允许重复修改，必要时可以提供恢复功能作为扩展。
5. 系统可以根据事项时间判断是否逾期。

### 4.3.4 今日事项与逾期事项

为了提升提醒能力，系统增加今日事项和逾期事项视图。

功能需求如下：

1. 展示今天需要处理的事项。
2. 展示已经超过事项时间但仍未完成的逾期事项。
3. 展示未来 7 天待处理事项。
4. 支持在首页数据看板中展示今日事项数量、逾期事项数量和待完成事项数量。

## 4.4 数据看板模块

数据看板用于展示联系人和事项的整体统计情况，帮助用户快速了解当前联系人管理状态。

功能需求如下：

1. 展示联系人总数。
2. 展示正常联系人数量。
3. 展示黑名单联系人数量。
4. 展示待完成事项数量。
5. 展示今日事项数量。
6. 展示逾期事项数量。
7. 展示已完成事项数量。
8. 展示联系人性别比例。
9. 展示未来 7 天事项趋势。
10. 支持点击看板卡片跳转到对应列表页面。

## 4.5 头像上传模块

头像上传模块用于管理用户头像和联系人头像。

功能需求如下：

1. 支持联系人头像上传。
2. 支持联系人头像预览。
3. 支持联系人头像更换。
4. 支持默认头像。
5. 支持限制上传文件类型，例如 jpg、png、jpeg、webp。
6. 支持限制上传文件大小。
7. 支持上传文件重命名，避免文件名冲突。
8. 支持数据库保存头像文件名或访问路径。
9. 支持前端通过静态资源路径访问头像。

## 4.6 Contact Agent 智能助手模块

Contact Agent 是系统的智能交互增强模块。它不替代原有表单功能，而是在基础链路完整可用的前提下，为用户提供自然语言操作入口。

### 4.6.1 模块定位

Contact Agent 主要用于处理联系人和事项相关的轻量任务，包括自然语言创建事项、查询联系人、查询事项、生成今日待办摘要和执行部分低风险操作。

系统需要保证 Agent 的操作边界清晰。对于查询类操作，可以直接返回结果；对于新增、取消、完成等写操作，需要用户二次确认后才能执行；对于批量修改等高风险操作，必须展示影响范围并等待用户确认。

### 4.6.2 自然语言创建事项

用户可以输入自然语言，系统自动解析联系人、时间和事项内容。

示例输入：

“下周三提醒我联系张三确认实习材料。”

系统解析结果：

1. 联系人：张三
2. 事项时间：下周三
3. 事项内容：联系张三确认实习材料
4. 事项状态：待完成

功能需求如下：

1. 支持从自然语言中解析联系人姓名。
2. 支持从自然语言中解析日期或时间。
3. 支持从自然语言中提取事项内容。
4. 如果联系人不存在，提示用户先创建联系人或手动选择联系人。
5. 如果存在多个同名联系人，要求用户选择具体联系人。
6. 在真正创建事项前展示解析结果并要求用户确认。
7. 用户确认后调用事项创建接口。
8. 创建成功后返回事项详情。

### 4.6.3 自然语言查询联系人

用户可以通过自然语言查询联系人。

示例输入：

“帮我找姓李的男生。”

系统解析条件：

1. 姓名以“李”开头。
2. 性别为男。
3. 联系人状态为正常。

功能需求如下：

1. 支持按姓名查询联系人。
2. 支持按电话查询联系人。
3. 支持按性别查询联系人。
4. 支持按标签查询联系人。
5. 支持组合条件查询联系人。
6. 查询结果以列表或卡片形式展示。
7. 查询结果支持跳转到联系人详情页。

### 4.6.4 自然语言查询事项

用户可以通过自然语言查询事项。

示例输入：

“我这周有哪些没完成的事情？”

系统解析条件：

1. 事项状态为待完成。
2. 事项时间在本周范围内。

功能需求如下：

1. 支持查询今日事项。
2. 支持查询本周事项。
3. 支持查询逾期事项。
4. 支持查询某个联系人的事项。
5. 支持根据事项状态查询。
6. 支持根据事项内容关键词查询。
7. 查询结果支持按时间排序。

### 4.6.5 今日待办总结

Contact Agent 可以根据用户的事项数据生成今日摘要。

功能需求如下：

1. 统计今日待完成事项数量。
2. 统计逾期未完成事项数量。
3. 提取最紧急的事项。
4. 输出简短摘要。
5. 支持跳转到今日事项列表。

示例输出：

“今天你有 3 个待完成事项，其中 1 个已经逾期。建议优先处理与张三相关的‘确认实习材料’事项。”

### 4.6.6 高风险操作二次确认

对于修改数据的操作，系统必须进行二次确认。

高风险操作包括：

1. 批量取消事项。
2. 批量完成事项。
3. 将联系人加入黑名单。
4. 恢复黑名单联系人。
5. 删除或覆盖头像。

处理流程如下：

1. Agent 解析用户意图。
2. Agent 查询受影响的数据范围。
3. 系统展示影响范围。
4. 用户确认。
5. 系统执行实际操作。
6. 系统返回操作结果。

## 5. 非功能性需求分析

## 5.1 安全性需求

1. 用户未登录时不能访问系统内部页面。
2. 用户只能访问本人账号下的联系人和事项数据。
3. 所有新增、修改、状态变更接口都需要校验当前用户身份。
4. 所有用户输入都需要进行前端和后端双重校验。
5. SQL 查询必须使用参数化方式，避免 SQL 注入。
6. 文件上传需要校验文件类型和文件大小。
7. 敏感日志中不能直接打印密码等隐私信息。
8. Contact Agent 不能绕过系统原有权限控制。
9. Contact Agent 的写操作必须经过用户确认。

## 5.2 可靠性需求

1. 系统应保证登录、联系人管理、事项管理等主流程稳定运行。
2. 数据库操作失败时需要回滚相关业务操作。
3. 图片上传失败时不能产生错误的联系人数据。
4. Agent 服务异常时，系统应降级到普通表单操作，不影响基础功能。
5. 后端应对空数据、空集合、非法 ID、重复操作等情况进行兜底处理。

## 5.3 易用性需求

1. 页面布局清晰，操作入口明确。
2. 表单控件选择合理，例如性别使用下拉框，日期使用日期选择器。
3. 错误提示应清晰可理解。
4. 联系人列表和事项列表支持分页，避免一次加载过多数据。
5. 移动端页面应适配窄屏浏览。
6. Contact Agent 的解析结果需要可编辑、可确认，避免误操作。

## 5.4 可维护性需求

1. 后端采用 Controller、Service、Mapper/DAO 分层结构。
2. 前端采用 views、components、api、router、stores 等模块化结构。
3. 公共响应结果统一封装。
4. 异常处理统一管理。
5. 常量、状态码、错误信息集中维护。
6. 复杂 SQL 独立维护，避免业务代码中大量拼接 SQL。
7. Contact Agent 工具调用应统一注册和管理。

## 5.5 兼容性需求

1. PC 端应支持 Chrome、Edge、Firefox 等主流浏览器。
2. 移动端应支持 Android 12 及以上系统浏览器。
3. 页面应采用响应式布局，在不同屏幕宽度下保持可用。
4. 后端接口应采用 RESTful JSON 格式，便于前后端联调。

## 6. 数据需求分析

系统主要涉及用户、用户头像、联系人、联系人头像、联系人事项等基础数据。为了支持扩展功能，可以增加联系人标签、Agent 操作记录等数据。

## 6.1 用户数据

用户数据用于登录认证和数据隔离。

主要字段包括：

1. 用户 ID
2. 用户密码
3. 用户头像
4. 创建时间
5. 更新时间

## 6.2 联系人数据

联系人数据是系统核心数据。

主要字段包括：

## 7. 业务流程分析

## 7.1 登录流程

1. 用户进入登录页面。
2. 用户输入账号和密码。
3. 前端进行非空校验。
4. 前端提交登录请求。
5. 后端查询用户信息。
6. 后端校验密码。
7. 校验成功后返回登录成功结果。
8. 前端保存登录状态并跳转到系统首页。
9. 校验失败则提示错误信息。

## 7.2 添加联系人流程

1. 用户进入添加联系人页面。
2. 用户填写联系人基础信息。
3. 用户选择头像文件。
4. 前端进行格式校验。
5. 前端提交联系人信息和头像文件。
6. 后端校验联系人字段。
7. 后端保存头像文件。
8. 后端保存联系人数据。
9. 后端保存联系人头像记录。
10. 系统返回添加成功结果。
11. 前端跳转到联系人列表页面。

## 7.3 加入黑名单流程

1. 用户在联系人列表中选择某个联系人。
2. 用户点击加入黑名单。
3. 系统弹出确认提示。
4. 用户确认后，前端提交请求。
5. 后端校验联系人是否属于当前用户。
6. 后端将联系人状态更新为黑名单。
7. 系统返回操作成功。
8. 联系人从正常联系人列表中移除。

## 7.4 恢复黑名单联系人流程

1. 用户进入黑名单列表。
2. 用户选择需要恢复的联系人。
3. 用户点击恢复按钮。
4. 系统弹出确认提示。
5. 用户确认后，前端提交请求。
6. 后端校验联系人是否属于当前用户。
7. 后端将联系人状态更新为正常。
8. 系统返回恢复成功。
9. 联系人重新出现在正常联系人列表中。

## 7.5 添加事项流程

1. 用户进入事项添加页面。
2. 用户选择联系人。
3. 用户选择事项时间。
4. 用户输入事项内容。
5. 用户选择事项优先级。
6. 前端校验输入内容。
7. 后端校验联系人归属和事项字段。
8. 后端保存事项数据。
9. 系统返回添加成功。
10. 前端跳转到事项列表页面。

## 7.6 Contact Agent 创建事项流程

1. 用户在智能助手输入自然语言。
2. 系统将用户输入发送到 Contact Agent。
3. Agent 识别用户意图为创建事项。
4. Agent 解析联系人、时间和事项内容。
5. 系统查询联系人是否存在。
6. 如果联系人不存在，则提示用户先添加联系人。
7. 如果存在多个同名联系人，则要求用户选择具体联系人。
8. 系统展示解析结果。
9. 用户确认。
10. 系统调用事项创建接口。
11. 系统返回创建成功结果。

用例名称：添加联系人  
参与者：普通用户

1. 用户进入添加联系人页面。
2. 用户填写联系人信息。
3. 用户上传联系人头像。
4. 用户点击添加按钮。

参与者：普通用户、Contact Agent

1. 用户输入自然语言事项描述。
2. Agent 识别创建事项意图。
3. Agent 解析联系人、时间和事项内容。
4. 系统展示解析结果。
5. 用户确认。
6. 系统创建事项。
7. 系统返回创建成功。

异常流程：

1. Agent 无法识别联系人，系统提示用户手动选择联系人。
2. Agent 无法识别时间，系统提示用户补充时间。
3. 存在多个同名联系人，系统要求用户选择。
4. AI 服务不可用，系统提示用户使用普通表单创建事项。

## 9. 系统边界

本系统主要解决联系人资料管理、事项提醒管理和轻量智能交互问题，不涉及短信发送、电话拨打、邮件自动发送、社交平台自动聊天等外部高风险能力。Contact Agent 只在系统内部执行受控工具调用，不能绕过权限直接修改数据库。

系统边界如下：

1. 系统不提供真实短信发送功能。
2. 系统不主动拨打电话。
3. 系统不自动发送邮件。
4. 系统不接入第三方社交账号。
5. Agent 不能直接执行未经用户确认的写操作。
6. Agent 服务失败不影响系统基础联系人和事项管理功能。

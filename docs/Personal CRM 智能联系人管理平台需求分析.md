# Personal CRM 智能联系人管理平台需求分析

## 1. 项目背景

随着个人学习、实习、社交和工作场景的不断扩展，用户需要维护的联系人数量逐渐增多，联系人之间的关系类型、沟通记录、待办事项和提醒任务也变得更加复杂。传统通讯录通常只支持姓名、电话、邮箱等基础信息的保存，缺少对联系人事项、关系标签、提醒任务和智能检索的统一管理能力，难以满足用户在学习、实习、求职和日常协作场景下的精细化联系人管理需求。

为解决上述问题，本项目设计并实现一套 Personal CRM 智能联系人管理平台。系统以联系人信息管理为核心，围绕联系人资料维护、黑名单管理、事项提醒、标签分类、搜索筛选、头像上传和移动端适配等功能，构建一个完整的在线信息管理系统。在此基础上，系统引入轻量级 Contact Agent 智能助手，支持自然语言创建事项、查询联系人、查询待办任务和生成今日事项摘要，从而提升系统的交互效率和使用体验。

## 2. 项目目标

本项目的总体目标是实现一个具备完整业务链路、良好交互体验和一定智能化能力的联系人管理系统。系统需要支持用户登录、联系人增删改查、联系人详情维护、黑名单管理、事项提醒管理、头像上传、搜索筛选排序分页、移动端适配等基础功能，并在基础功能稳定运行的前提下，扩展自然语言交互能力。

具体目标如下：

1. 实现用户注册与基于邮箱验证码的账号激活，支持找回密码及在登录态下修改绑定邮箱，确保数据物理隔离。
2. 实现联系人信息的新增、查询、修改、屏蔽（黑名单）、恢复与逻辑删除等完整管理能力。
3. 实现联系人与个人头像上传、预览和生命周期维护。
4. 实现联系人事项的添加、查询、取消、完成、逻辑删除和状态筛选，支持动态判定逾期。
5. 实现联系人、黑名单和事项的多维搜索、标签筛选、时间排序与分页。
6. 实现 PC 浏览器端和移动端浏览器的响应式适配与极佳的用户交互体验。
7. 基于 Redis 二级缓存实现高性能数据看板，展示联系人总数、黑名单数、待完成数、今日及逾期事项，并提供未来 7 天事项趋势和关系健康度多维分析。
8. 实现基于 OpenAI 兼容大模型的 Contact Agent，支持多轮澄清、槽位填充、Redis 分布式会话状态托管以及涉及写操作的二次确认。
9. 实现联系人活动轨迹（Activity Feed）自动留痕，支持在联系人详情页以时间线形式展示其创建、修改、标签变更、黑名单切换及事项关联状态。
10. 集成第三方天气服务代理，支持首页显示本地天气与联系人详情页展示近 3 天关怀天气预报（含 IP/GEO 多级定位回退及 2 小时结果缓存）。
11. 保证系统分层架构清晰、代码具有良好的可读性，具备全面的异常捕获和精细化的状态反馈。

## 3. 用户角色分析

本系统主要面向单个普通用户使用，用户登录后只能管理本人账号下的联系人和事项数据。系统角色相对简单，主要包括以下两类：

### 3.1 普通用户

普通用户是系统的主要使用者，拥有联系人管理、事项管理、黑名单管理、头像上传、智能助手使用等权限。

普通用户可以完成以下操作：

1. 注册账号并接收邮箱验证码完成激活。
2. 忘记密码时通过邮箱验证码进行找回和重置。
3. 登录系统并获取当前用户的详细个人资料，并在登录状态下通过邮箱验证更换绑定邮箱。
4. 查看正常联系人列表和黑名单列表。
5. 添加新的联系人（支持绑定标签）。
6. 查看联系人详情（含活动日志时间线及关怀天气预报）。
7. 修改联系人资料并支持逻辑删除。
8. 上传或更换用户头像与联系人头像。
9. 将联系人加入黑名单或从中恢复。
10. 查看联系人相关事项列表。
11. 添加联系人待办事项（支持设置优先级）。
12. 变更待办事项状态为已完成或已取消，或者进行逻辑删除。
13. 查看首页看板，包含个人统计指标、未来 7 天事项趋势、性别比例图、关系维护健康度多维分析以及本地天气。
14. 使用 Contact Agent 进行自然语言交互：支持查询联系人/事项、多轮澄清槽位追问、今日待办口语化总结。
15. 在 Contact Agent 生成事项创建等写操作时，进行确认执行或取消。
16. 通过页面中的电话和邮箱的原生协议链接快捷唤起外部系统拨号或发信。

### 3.2 系统管理员（可选扩展）

如果项目需要进一步扩展，可以增加系统管理员角色。系统管理员主要负责演示数据维护、用户管理和系统配置管理。但在本次实习项目中，管理员角色不是核心要求，可以作为后续扩展功能。

## 4. 功能性需求分析

## 4.1 认证、注册与邮箱安全验证模块

该模块是系统的入口和账号安全边界。系统支持基于账号密码的登录，并提供了完整的用户注册、注册邮箱激活、忘记密码通过邮箱重置，以及登录态下修改绑定邮箱的安全闭环。系统在后端依托 Redis 提供了防刷限流与防暴力破解的爆破防护。

### 4.1.1 核心功能需求

1. **用户注册**：用户可以提供用户名、密码和邮箱进行注册。新注册的账号状态默认为“待验证 (UNVERIFIED)”，必须通过邮箱验证码进行激活才能登录。
2. **邮箱验证码发送**：支持向用户指定的邮箱发送 6 位数字的临时安全验证码，可用于注册激活 (REGISTER)、密码重置 (RESET_PASSWORD) 和修改邮箱 (CHANGE_EMAIL)。
3. **验证码安全与限流防护**：
   - **频率锁**：对同一邮箱、同一用途的验证码发送，设置 60 秒的冷却周期（防刷频）。
   - **有效期**：验证码生命周期为 300 秒，逾期自动失效。
   - **爆破锁定锁**：对同一邮箱、同一用途 of 验证码校验失败尝试，在 300 秒内上限为 5 次。若输错超过 5 次，锁定该用途的校验，验证码作废。
4. **账号激活**：用户输入收到的 6 位数字验证码，校验成功后，将账号状态由“待验证”置为“激活 (ACTIVE)”，并允许正常登录。
5. **用户登录**：已激活的用户通过账号和密码登录，登录成功后后端颁发 JWT Token。未登录用户拦截并返回 401 状态码。
6. **找回/重置密码**：在未登录状态下，用户可通过输入注册邮箱、邮箱验证码以及新密码来重置密码。
7. **修改绑定邮箱**：已登录用户可在个人中心提供新邮箱，并输入新邮箱收到的验证码以更换绑定的邮箱。

### 4.1.2 输入校验

1. 用户名、密码和邮箱不能为空。
2. 邮箱地址必须符合标准的电子邮箱格式。
3. 验证码必须为 6 位纯数字。
4. 新密码应符合系统的复杂度要求（至少 6 位）。

### 4.1.3 异常与安全处理

1. 账号不存在或密码错误时，返回模糊的“用户名或密码错误”提示，避免恶意探测。
2. 邮箱被重复注册、或修改时新邮箱已被占用，返回“该邮箱已被注册”冲突异常。
3. 验证码发送冷却中（60秒内），返回“发送验证码过于频繁，请稍后再试”。
4. 验证码输错 5 次被锁，提示“验证码尝试次数过多，已被锁定，请重新发送验证码”。
5. JWT Token 过期或篡改时，全局拦截器自动返回 401 状态码。

## 4.2 联系人管理模块

联系人管理模块是系统核心模块，用于维护用户的联系人资料。联系人信息包括姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日、头像和联系人状态等。

### 4.2.1 联系人列表

联系人列表用于展示当前用户的正常联系人。用户可以在该页面查看联系人基本信息，并进行搜索、筛选、排序、分页、查看详情、添加联系人、加入黑名单和进入事项页面等操作。

功能需求如下：

1. 展示当前登录用户的正常联系人。
2. 支持分页展示联系人。
3. 支持根据姓名或手机号搜索联系人。
4. 支持根据性别筛选联系人。
5. 支持根据姓名、生日或创建时间进行排序。
6. 支持跳转到联系人详情页。
7. 支持跳转到添加联系人页面。
8. 支持将联系人加入黑名单。
9. 支持跳转到联系人事项页面。
10. 支持展示联系人头像、姓名、性别、电话等核心信息。

### 4.2.2 添加联系人

添加联系人页面用于录入新的联系人信息。用户填写联系人姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日等信息后，可以提交保存。

功能需求如下：

1. 支持输入联系人姓名。
2. 支持选择联系人性别。
3. 支持输入联系人电话。
4. 支持输入联系人邮箱。
5. 支持输入联系人 QQ。
6. 支持输入联系人微信。
7. 支持输入联系人地址。
8. 支持输入联系人邮编。
9. 支持选择联系人生日。
10. 支持上传联系人头像。
11. 支持添加成功后跳转回联系人列表页。
12. 支持添加失败时提示失败原因。

### 4.2.3 联系人详情

联系人详情页面用于查看和修改联系人完整资料。

功能需求如下：

1. 展示联系人姓名、性别、电话、邮箱、QQ、微信、地址、邮编、生日和头像。
2. 支持修改联系人资料。
3. 支持更换联系人头像。
4. 支持保存修改结果。
5. 支持跳转到该联系人的事项列表。
6. 支持返回联系人列表。

### 4.2.4 联系人黑名单

黑名单用于保存被屏蔽的联系人。加入黑名单后，该联系人不再出现在正常联系人列表中，但数据不会被物理删除。

功能需求如下：

1. 支持将正常联系人加入黑名单。
2. 支持查看黑名单联系人列表。
3. 支持搜索黑名单联系人。
4. 支持根据性别筛选黑名单联系人。
5. 支持分页展示黑名单联系人。
6. 支持查看黑名单联系人详情。
7. 支持将黑名单联系人恢复为正常联系人。

### 4.2.5 联系人标签扩展

为了增强 Personal CRM 的关系管理能力，系统可以为联系人增加标签功能。标签可以用于标识联系人关系类型，例如同学、老师、家人、朋友、客户、重要联系人等。

功能需求如下：

1. 支持创建联系人标签。
2. 支持给联系人绑定一个或多个标签。
3. 支持根据标签筛选联系人。
4. 支持在联系人详情中展示联系人标签。
5. 支持修改联系人标签。

该功能属于增强功能，在基础链路完成后实现。

### 4.2.6 联系人逻辑删除

用户可以在联系人列表或详情页中删除联系人。为了维护历史关联数据一致性与数据安全，系统在此处实行逻辑删除（Soft Delete）策略。

功能需求如下：
1. **租户归属限制**：仅限当前登录用户可以删除自己拥有的联系人，若操作越权数据，后端统一拦截并返回 403 权限错误。
2. **状态标记**：逻辑删除只在数据库中将该联系人的逻辑删除标志 `deleted` 置为 1，不物理清除该条数据。
3. **级联清理规则**：删除联系人时，后端自动级联清理该联系人与标签的绑定关系（在 `contact_tag` 表中删除对应记录），避免脏数据残留；同时，该联系人历史已创建的事项（Todo）予以完好保留。
4. **可见性屏蔽**：被逻辑删除的联系人，不再显示在正常列表或黑名单列表中，也不会在任何查询、看板统计及 Agent 交互结果中被召回。

## 4.3 联系人事项提醒模块

联系人事项提醒模块用于管理与联系人相关的待办事项。事项包括事项时间、事项内容、事项状态和所属联系人。

### 4.3.1 事项列表

事项列表用于展示联系人相关事项或当前用户所有事项。

功能需求如下：

1. 支持查看当前用户的所有事项。
2. 支持查看某个联系人的所有事项。
3. 支持根据事项内容搜索事项。
4. 支持根据事项状态筛选事项。
5. 支持根据事项时间升序或降序排序。
6. 支持分页展示事项。
7. 支持将事项标记为已完成。
8. 支持将事项标记为已取消。
9. 支持跳转到添加事项页面。
10. 支持展示事项所属联系人、事项时间、事项内容和事项状态。

### 4.3.2 添加事项

添加事项页面用于为联系人添加新的提醒事项。

功能需求如下：

1. 支持选择联系人。
2. 支持选择事项时间。
3. 支持输入事项内容。
4. 支持设置事项优先级。
5. 支持提交后保存为待完成事项。
6. 支持添加成功后返回事项列表。
7. 支持输入校验和错误提示。

### 4.3.3 事项状态管理

事项状态包括待完成、已取消和已完成。

功能需求如下：

1. 新增事项默认状态为待完成。
2. 用户可以将待完成事项标记为已完成。
3. 用户可以将待完成事项标记为已取消。
4. 已完成或已取消事项默认不允许重复修改，必要时可以提供恢复功能作为扩展。
5. 系统可以根据事项时间判断是否逾期。

### 4.3.4 今日事项与逾期事项

为了提升提醒能力，系统增加今日事项和逾期事项视图。

功能需求如下：

1. 展示今天需要处理的事项。
2. 展示已经超过事项时间但仍未完成的逾期事项。
3. 展示未来 7 天待处理事项。
4. 支持在首页数据看板中展示今日事项数量、逾期事项数量和待完成事项数量。

### 4.3.5 事项逻辑删除

用户可以在事项列表中将不需要的事项进行删除。为了维护历史轨迹数据安全，系统实行逻辑删除（Soft Delete）策略。

功能需求如下：
1. **租户归属限制**：仅限当前登录用户可以删除自己拥有的事项。
2. **状态标记**：逻辑删除只在数据库中将该事项的逻辑删除标志 `deleted` 置为 1。
3. **可见性屏蔽**：被逻辑删除的事项将不再出现在任何事项列表、今日代办统计或 Agent 交互结果中。

## 4.4 数据看板模块

数据看板用于展示联系人、事项和关系维护的整体统计情况，通过图形可视化（ECharts）帮助用户直观、快速了解当前的个人客户管理状态。

功能需求如下：

1. **核心数据卡片**：
   - 展示联系人总数（排除逻辑删除）。
   - 展示正常联系人与黑名单联系人数量。
   - 展示待完成事项、今日事项和逾期事项的数值统计。
2. **可视化图表**：
   - **性别比例图**：使用饼图展现联系人中的性别分布。
   - **未来 7 天事项趋势图**：展现接下来一星期的待办事项数量走势。
   - **关系维护健康度多维分析**：多维度展示联系人关系健康度状况（活跃、随访、流失等状态比例）。
3. **缓存优化**：
   - 看板数据在后端引入 Redis 二级缓存，以减少数据库压力。当数据发生新增、修改或删除（逻辑删除）等写操作时，自动清空缓存以保证前后端数据一致性。
4. **前端性能控制**：
   - ECharts 图表组件在路由切换或组件销毁时，需显式销毁并释放实例（置为 null），防止内存泄露。
5. **快捷跳转**：
   - 支持点击看板统计卡片，自动跳转至对应的联系人列表、黑名单列表或事项列表页面。

## 4.5 头像上传模块

头像上传模块用于管理用户头像和联系人头像。

功能需求如下：

1. 支持联系人头像上传。
2. 支持联系人头像预览。
3. 支持联系人头像更换。
4. 支持默认头像。
5. 支持限制上传文件类型，例如 jpg、png、jpeg、webp。
6. 支持限制上传文件大小。
7. 支持上传文件重命名，避免文件名冲突。
8. 支持数据库保存头像文件名或访问路径。
9. 支持前端通过静态资源路径访问头像。

## 4.6 活动日志模块 (Activity Feed)

活动日志模块用于自动、异步或同步地留痕联系人及事项的相关业务操作，并在前端展示为时间线（Timeline），方便用户跟踪其互动轨迹。

功能需求如下：
1. **自动触发留痕**：
   - 当用户创建联系人、修改联系人资料、变更其标签绑定、切换黑名单状态时，系统自动生成相应日志。
   - 当用户创建与该联系人关联的事项，或将事项完成、取消时，系统自动关联生成事项动态日志。
2. **多租户隔离查询**：
   - 轨迹日志由 `ActivityLogService` 统一管理，查询时必须联合当前用户的 `userId` 和 `contactId`，防止水平越权。
3. **前端时间线展现**：
   - 联系人详情页展示“最近动态”卡片。按发生时间倒序展现该联系人的操作足迹（最多10条），不同类型的活动配置不同的图标与提示文本（如“创建了联系人”、“完成了待办事项”）。
   - 若联系人没有操作轨迹，显示友好的空状态。

## 4.7 天气代理服务与关怀组件

系统集成第三方天气服务接口，提供用户本地与联系人所在城市的天气预报，作为“关怀问候”沟通支持。

功能需求如下：
1. **首页本地天气**：
   - 看板根据登录用户的请求 IP（或浏览器定位 GEO 坐标）识别其所在城市，展示当前温度、天气状况及图标，并显示日期与相应问候语（“早上好”、“下午好”）。
2. **联系人关怀天气**：
   - 在联系人详情页中，系统解析该联系人填写的“地址”信息，换取其城市编码，展示接下来 3 天的天气预报（支持折叠展开）。
   - 当联系人地址缺失时，提示并提供一键引导补全入口。
3. **定位回退与性能优化**：
   - 定位解析采用 `模糊地址 > 浏览器 GEO > 客户端 IP > 杭州` 的多级降级机制，保证总能返回天气信息。
   - 后端对天气查询结果引入 2 小时内存缓存，节约 API 请求配额并提升加载效率。

## 4.8 Contact Agent 智能助手模块

Contact Agent 是系统的智能交互增强模块。它不替代原有表单功能，而是在基础链路完整可用的前提下，为用户提供便捷的自然语言操作入口。本模块通过外部 OpenAI 兼容大模型进行意图识别与槽位填充，并基于 Redis 进行多轮会话状态托管。

### 4.8.1 模块定位与安全边界

Agent 模块不直接操作数据库，而是通过受控的 Tool 调用系统已有的业务 Service，复用系统的安全鉴权、参数校验与事务管理逻辑。
- 对于**查询类请求**，Agent 可直接解析条件并返回数据；
- 对于**创建事项等写操作**，必须由用户通过确认流进行二次确认后方可落库。

### 4.8.2 自然语言创建事项与多轮澄清

用户输入自然语言，系统自动识别并提取 `联系人姓名`、`时间`、`内容` 和 `优先级`。
- **槽位填充与多轮澄清**：如果信息缺失（如未指定联系人或时间），Agent 不会直接报错，而是通过 `sessionId` 维持上下文，并在前端向用户发起澄清追问（例如：“请问您要为哪位联系人创建该事项？”）。
- **同名消歧**：如果输入姓名匹配到多个联系人，Agent 将列出同名联系人要求用户选择。
- **预确认卡片与二次确认**：当所有要素收集齐备后，系统生成带有结构化参数的“预确认卡片”，用户点击“确认执行”后，系统调用后台 TodoService 落库并向用户返回成功卡片；用户点击“取消”则该日志作废。

### 4.8.3 自然语言多维查询

用户可以使用口语化的方式进行联系人和事项的搜索。
- **联系人查询**：支持按姓名、手机号、微信、性别、标签等单一或组合条件搜索，结果以卡片形式展示，支持一键查看详情。
- **事项查询**：支持查询今日待办、本周待办、逾期事项，或某一特定联系人的事项，并按时间倒序排列。

### 4.8.4 今日待办口语化总结

支持用户输入“今天有什么事”、“总结一下今日待办”等指令，Agent 统计今日待办、逾期未完成事项，并挑出最紧急事项进行段落式润色口语化总结，提供快速跳转到事项列表的链接。

### 4.8.5 分布式会话状态托管

Agent 的多轮澄清上下文状态（槽位和对话历史）以 JSON 格式序列化并保存在 Redis 中（Key: `agent:session:{sessionId}`），TTL 设置为 600 秒（10分钟），过期自动销毁，实现后端无状态与自愈。

## 5. 非功能性需求分析

## 5.1 安全性需求

1. 用户未登录时不能访问系统内部页面。
2. 用户只能访问本人账号下的联系人和事项数据。
3. 所有新增、修改、状态变更接口都需要校验当前用户身份。
4. 所有用户输入都需要进行前端和后端双重校验。
5. SQL 查询必须使用参数化方式，避免 SQL 注入。
6. 文件上传需要校验文件类型和文件大小。
7. 敏感日志中不能直接打印密码等隐私信息。
8. Contact Agent 不能绕过系统原有权限控制。
9. Contact Agent 的写操作必须经过用户确认。

## 5.2 可靠性需求

1. 系统应保证登录、联系人管理、事项管理等主流程稳定运行。
2. 数据库操作失败时需要回滚相关业务操作。
3. 图片上传失败时不能产生错误的联系人数据。
4. Agent 服务异常时，系统应降级到普通表单操作，不影响基础功能。
5. 后端应对空数据、空集合、非法 ID、重复操作等情况进行兜底处理。

## 5.3 易用性需求

1. 页面布局清晰，操作入口明确。
2. 表单控件选择合理，例如性别使用下拉框，日期使用日期选择器。
3. 错误提示应清晰可理解。
4. 联系人列表和事项列表支持分页，避免一次加载过多数据。
5. 移动端页面应适配窄屏浏览。
6. Contact Agent 的解析结果需要可编辑、可确认，避免误操作。

## 5.4 可维护性需求

1. 后端采用 Controller、Service、Mapper/DAO 分层结构。
2. 前端采用 views、components、api、router、stores 等模块化结构。
3. 公共响应结果统一封装。
4. 异常处理统一管理。
5. 常量、状态码、错误信息集中维护。
6. 复杂 SQL 独立维护，避免业务代码中大量拼接 SQL。
7. Contact Agent 工具调用应统一注册和管理。

## 5.5 兼容性需求

1. PC 端应支持 Chrome、Edge、Firefox 等主流浏览器。
2. 移动端应支持 Android 12 及以上系统浏览器。
3. 页面应采用响应式布局，在不同屏幕宽度下保持可用。
4. 后端接口应采用 RESTful JSON 格式，便于前后端联调。

联系人数据是系统的核心业务数据。

## 6.3 联系人头像数据

联系人头像数据用于保存上传的图片文件元数据。

主要字段包括：
1. 图片 ID（业务编号）

联系人事项数据用于保存与联系人相关的日程提醒待办。

联系人标签属于增强分类数据。

主要字段包括：

## 7. 业务流程分析

## 7.1 登录流程

1. 用户进入登录页面。
2. 用户输入账号和密码。
3. 前端进行非空校验。
4. 前端提交登录请求。
5. 后端查询用户信息。
6. 后端校验密码。
7. 校验成功后返回登录成功结果。
8. 前端保存登录状态并跳转到系统首页。
9. 校验失败则提示错误信息。

## 7.2 添加联系人流程

1. 用户进入添加联系人页面。
2. 用户填写联系人基础信息。
3. 用户选择头像文件。
4. 前端进行格式校验。
5. 前端提交联系人信息和头像文件。
6. 后端校验联系人字段。
7. 后端保存头像文件。
8. 后端保存联系人数据。
9. 后端保存联系人头像记录。
10. 系统返回添加成功结果。
11. 前端跳转到联系人列表页面。

## 7.3 加入黑名单流程

1. 用户在联系人列表中选择某个联系人。
2. 用户点击加入黑名单。
3. 系统弹出确认提示。
4. 用户确认后，前端提交请求。
5. 后端校验联系人是否属于当前用户。
6. 后端将联系人状态更新为黑名单。
7. 系统返回操作成功。
8. 联系人从正常联系人列表中移除。

## 7.4 恢复黑名单联系人流程

1. 用户进入黑名单列表。
2. 用户选择需要恢复的联系人。
3. 用户点击恢复按钮。
4. 系统弹出确认提示。
5. 用户确认后，前端提交请求。
6. 后端校验联系人是否属于当前用户。
7. 后端将联系人状态更新为正常。
8. 系统返回恢复成功。
9. 联系人重新出现在正常联系人列表中。

## 7.5 添加事项流程

1. 用户进入事项添加页面。
2. 用户选择联系人。
3. 用户选择事项时间。
4. 用户输入事项内容。
5. 用户选择事项优先级。
6. 前端校验输入内容。
7. 后端校验联系人归属和事项字段。
8. 后端保存事项数据。
9. 系统返回添加成功。
10. 前端跳转到事项列表页面。

## 7.6 Contact Agent 创建事项流程

1. 用户在智能助手输入自然语言。
2. 系统将用户输入发送到 Contact Agent。
3. Agent 识别用户意图为创建事项。
4. Agent 解析联系人、时间和事项内容。
5. 系统查询联系人是否存在。
6. 如果联系人不存在，则提示用户先添加联系人。
7. 如果存在多个同名联系人，则要求用户选择具体联系人。
8. 系统展示解析结果。
9. 用户确认。
10. 系统调用事项创建接口。
11. 系统返回创建成功结果。

## 9. 系统边界

本系统主要解决个人联系人资料管理、事项提醒和轻量智能交互问题，不涉及短信发送、电话拨打、社交平台自动聊天等外部高风险能力。Contact Agent 只在系统内部执行受控工具调用，不能绕过权限直接修改数据库。

系统边界如下：

1. 系统不提供真实短信发送功能。
2. 系统不主动拨打电话（前端仅提供原生拨号 `tel:` 协议包装，由设备外部系统处理）。
3. 系统仅在用户注册激活、重置密码和更换邮箱时，通过外部发信服务（Resend SDK）发送账户验证通知邮件，不提供用户通用的邮件收发和发送功能。
4. 系统不接入第三方社交账号（仅保存其微信/QQ作为静态字段说明）。
5. Agent 不能直接执行未经用户确认的写操作。
6. Agent 服务失败不影响系统基础联系人和事项管理功能。

## 10. 验收标准

系统完成后，应满足以下验收标准：

1. **用户认证与安全验收**：
   - 用户可以注册新账号，注册时接收真实的邮箱验证码并激活账号。
   - 用户可以进行正常登录，获取 Token。
   - 用户可在登录态下进行邮箱更换，未登录时通过邮箱重置密码。
   - 安全验证码支持 Redis 频率限制、有效期控制与输错爆破防范锁定。
2. **联系人管理验收**：
   - 用户可以查看正常联系人列表和黑名单列表，支持分页、模糊搜索、性别筛选、生日/创建时间排序。
   - 支持新增、修改、拉黑与恢复联系人。
   - 支持联系人的逻辑删除，删除时在应用层级联清理标签绑定关联。
3. **事项管理验收**：
   - 支持新增事项，设置优先级。
   - 支持待完成、已取消、已完成的状态流转，已完成/已取消状态不可逆转。
   - 支持事项的逻辑删除，支持动态判断事项逾期。
4. **数据看板验收**：
   - 能统计联系人、黑名单、待办、今日、逾期数量，展现性别饼图与事项趋势图。
   - 支持关系维护健康度多维分析（活跃、随访、流失等状态比例）。
   - 首页看板支持 Redis 二级缓存，写操作时被动清空以实现强一致性。
   - 前端图表在组件卸载时需完全销毁释放图表实例以避免内存泄漏。
5. **上传与天气组件验收**：
   - 支持联系人头像和用户头像文件大小（2MB内）与格式校验。
   - 首页看板展示用户本地天气（IP/GEO自动回退）；联系人详情页展示其地址对应的 3 天天气预报，支持 2 小时结果缓存。
6. **活动轨迹验收**：
   - 核心写操作自动留痕，可在联系人详情页中以时间轴倒序展现其创建、变更、标签绑定、事项操作等最新轨迹。
7. **Contact Agent 智能交互验收**：
   - 具备独立页面进行联系人、待办的口语化多维查询。
   - 能够解析意图并提取槽位，缺少槽位时支持通过 Redis 多轮会话进行追问澄清，存在同名时支持消歧选择。
   - “创建事项”等写操作必须输出确认卡片，经用户确认后方可调用 TodoService 落库。
   - AI 服务不可用时，系统降级，普通表单功能仍然可以正常运行。

## 11. 本章小结

本章从项目背景、项目目标、用户角色、功能性需求、非功能性需求、数据需求、业务流程、用例分析、系统边界和验收标准等方面，对 Personal CRM 智能联系人管理平台进行了完整需求分析。系统在满足联系人管理、黑名单管理、事项提醒、头像上传、分页搜索排序和移动端适配等基础要求的同时，进一步引入 Contact Agent 智能助手，实现自然语言创建事项、自然语言查询和今日待办总结等增强能力。该设计既保证了在线信息系统基础链路的完整性，也提升了系统的智能化水平和实际应用价值。
